# PRD 审阅检查清单

## 审阅原则

**只报告真正的问题**，不要鸡蛋里挑骨头。如果某项没有明显问题，跳过它。

🔴 **必须修复**：会导致开发无法进行或产生严重误解的问题
🟡 **建议优化**：明显的遗漏或不一致，但不阻塞开发

不报告的情况：
- 格式/措辞的细微差异
- 理论上可能但实际不太会发生的边界情况
- "可以更好"但当前已经够用的内容
- 主观的风格偏好

## 🔴 必须修复的问题类型

### 需求不可执行
- 需求描述模糊到开发无法理解要做什么
- 关键约束缺失（如字段长度、数值范围完全未提及且会影响实现）

### 逻辑矛盾
- 同一文档内规则互相冲突
- 角色权限与功能需求明显矛盾
- 状态流转存在死循环或无法到达的状态

### 关键遗漏
- 核心 CRUD 操作缺失（如有创建但完全没提删除，且删除是必需的）
- 数据结构缺少支撑核心功能的必要字段

## 🟡 建议优化的问题类型

### 明显的不完整
- 提到了某角色但未说明其权限
- 业务规则提到的状态在功能需求中未定义

### 跨功能不一致
- 同一实体在不同功能中字段定义冲突
- 功能间依赖关系未说明

## 不需要报告

- MVP 范围划分的合理性（这是产品决策）
- 是否应该添加更多功能（除非用户问）
- 措辞是否可以更精确（只要能理解就行）
- 是否缺少某些"最佳实践"（如软删除、审计日志等，除非用户要求）
